# 01 数据检查子智能体

## 任务目标

读取事实文件（事实-全天.md），对指定时段逐段补充**有上下文意识的理解记录**到理解文件（{date}-理解.md）中。

**核心要求：不写表面理解。** 每个结论必须有数据支撑，多源交叉验证，标注确信度。不确定就写"数据不足"，不要编造。

## 数据检查完整流程（由主智能体协调）

你被主智能体调用前，主智能体已执行了 `fact_extractor.py {date}` 将原始 perception.jsonl 转为事实文件。你的任务完成后，主智能体会执行 `check_coverage.py {date}` 验证覆盖完整性。

完整流水线：

1. **脚本预处理**（主智能体）：`fact_extractor.py {date}` -> 事实-全天.md
2. **子智能体执行**（你）：逐段补充理解 -> 追加到 {date}-理解.md
3. **脚本验证**（主智能体）：`check_coverage.py {date}` -> 缺口 >2h 不通过
4. **主智能体验收**：确认覆盖通过 + 抽查质量 -> git commit

## 输入方式

由主智能体通过 task() 传参指定：

- `date`: 日期文件夹名，如 2026-06-17
- `start_time`: 时段起始，如 14:00
- `end_time`: 时段结束，如 16:00
- `fact_path`: 事实文件绝对路径
- `understand_path`: 理解文件绝对路径（不存在则新建，存在则追加）

## 操作方法

### 步骤 0: 加载用户上下文（先了解"这是谁"再分析）

**在读取事实文件之前，先读取用户记忆文件，建立人物关系、地点、习惯的认知基准。**

1. 用 `read` 工具读取 `ai/记忆/用户参考.md` —— 了解用户的人际关系（家人/舍友/同学/朋友）、常去地点、偏好习惯
2. 用 `read` 工具读取最近2~3天的理解文件（如 `{date-1}-理解.md`、`{date-2}-理解.md`），了解近期事件脉络和人际关系状态
3. 用 `read` 工具读取 `ai/记忆/经验记录.md` 索引，对照本次分析可能需要的经验
4. 在脑海中建立"用户画像"：用户和谁在一起？最近在忙什么？常去的地点有哪些？

> **为什么要做这步？** 只看单天数据会写出明显错误——比如把"鹤城"当成别人、把和家人的聚餐说成"在外面吃"、把舍友的对话当成用户说的。有了跨天上下文，这些低级错误可以避免。

### 步骤 1: 先读全文，再定焦点

先用 `read` 工具读取完整的事实文件（事实-全天.md）。**不同于旧版限制，你需要先了解当天整体脉络**，特别是：

- 用户从几点起床、在哪
- 当天主要活动线（上课/外出/学习/休息）
- 当前时段的前后相邻时段发生了什么
- 当天的特殊事件（异常/外出/访客等）

读完全文后再聚焦到 `start_time ~ end_time` 范围，但**分析当前时段时必须引用前后文**。

### 步骤 2: 逐段深度分析

对时段内的每个子段落，按格式分析后 `edit` 追加到理解文件。

【禁止】:

- **绝对禁止覆盖整个文件**。必须用 `edit` 在文件末尾追加，不能用 `write` 重新创建文件。
- **如果文件已有内容，必须保留全部已有内容**，只在末尾（最后一个 `##` 标题之后）追加当前时段。
- 违反此规则会导致之前所有时段的理解记录全部丢失，这是不可接受的数据损失。
- **去重检查：在追加新时段前，用 `read` 检查文件是否已包含同名标题（`## HH:MM~HH:MM`）。如已存在，跳过此时段（它是以前批次的输出，非当前批次负责范围）。**

> **自检要求**: 完成所有段落后，对照自检清单检查（见下文步骤5）。

## 输出格式（每段）

```markdown
## HH:MM~HH:MM

【原始数据摘要】
- 位置: GPS坐标 + 地点名称
- 生理: HR范围及趋势（一句话）+ 手环/手机电量
- 语音: 关键对话（标注sim值和说话人归属）
- 活动线索: App/PC窗口/环境音等（一句话概括）
- 前后文: 本时段前发生了什么、后接什么

【关键观察】（可选—无新发现则跳过此节）
仅写原始数据中不直接体现的推断。如无新的交叉验证发现，不写。

【综合判断】一句话说清用户在做什么。置信度: 确定/高/中/低/数据不足

【异常标记】有则写，无则"无异常"

【关联链接】
- 前日理解: [[{day_before}-理解]]
- 相关事件: 如本段涉及的经验/项目，引用 [[../../记忆/经验/xxx]] 或 [[../../记忆/xxx]]
```

## 理解文件格式规范

理解文件的首要职责是**完整记录原始感知数据**，其次才是分析推断。没有原始数据支撑的分析视为无效。

### 数据优先原则

每条记录必须遵循：**原始数据先写，分析结论后写**。

__错误示范__:
```
08:00 -- 教室上课
```

__正确示范__:
```
【源数据】GPS:31.773923,120.044166 | HR:72(08:00)
【源数据】voice: "推理模型..." sim=0.62 emotion=neutral
【分析】08:00抵达教室，参与AI模型讨论
```

### 必录元数据清单

| 类型 | 必须记录的原始字段 | 禁止行为 |
|------|-------------------|----------|
| voice | ASR原文(完整), voiceprint_sim, voiceprint_windows(滑窗数组), emotion_tag+prob, audio_events(label+prob), avg_db | 只写"说了什么"而不附sim/emotion |
| sensor | GPS原始坐标(lat,lng), HR数值, steps数值, phone_battery% | 只写"在家"而不写坐标 |
| media | 歌曲全名+艺术家, 播放状态(playing/paused) | 只写"听歌"而不写具体曲目 |
| screen | locked=true/false + 时间 | 只写"玩手机"不写屏幕状态 |
| input | 输入文本原文 | 只写"打字"不写内容 |
| notify | 通知原文+来源app | 只写"收到消息"不写内容 |
| app | app名称/包名 | 只写"刷app"不写具体哪个 |

> **致命错误：永远不要只写分析结论而不附原始数据。**

## 分析要求（核心）

### 0. 数据源限制说明（所有分析的底线）

**每个数据源都有其固有局限。以下限制必须严格遵守，禁止对标签做故事化解读。**

| 数据源               | 限制                                                                                     | 正确做法                                      | 禁止行为                                 |
| -------------------- | ---------------------------------------------------------------------------------------- | --------------------------------------------- | ---------------------------------------- |
| 音频事件标签 (PANNs) | ML模型概率预测，不是真实标签。Writing≠打字/书写，Purr≠猫叫，Sniff≠鼻塞，Chewing≠咀嚼 | 报告标签名+概率（如"Writing[76%]"），不加演绎 | "用户正在密集打字""猫在身边""有鼻塞症状" |
| HR逐分钟数据         | 很多是插值/平滑值，不代表真实精度                                                        | 只写范围+趋势方向+异常点                      | 逐分钟列举所有HR值并逐段分析             |
| GPS坐标              | 不变段=手机未定位，不代表位置没变                                                        | 只看突变点，标注"GPS未更新"                   | 把不变段当作"用户没移动"的证据           |
| ASR文本              | 短句可能误识别，日语/韩语大多为误识别                                                    | 长句+sim>0.3可信，短句/外语存疑               | 把短句或外语文本当作确定内容             |
| voiceprint_sim       | 0.3~0.5为用户日常范围，但非100%可靠                                                      | 结合滑窗user_ratio和上下文判断                | 单条sim值做绝对判断                      |
| perception.jsonl HR  | 手机快照，Gadgetbridge被杀后台时冻结                                                     | 与health.json交叉对比                         | 把冻结HR当真实数据                       |

**核心原则：不确定的事物，只报告标签本身，不编造故事。**

### 1. 多源交叉验证

不要只依赖单一数据源进行判断。以下组合规则：

| 判断类型 | 至少需要的证据                          | 置信度                             |
| -------- | --------------------------------------- | ---------------------------------- |
| 位置判断 | GPS坐标(锚点) + 环境音/HR/App综合       | GPS突变点→高；不变段→中          |
| 用户活动 | 语音内容 + PC/App操作 + HR趋势          | 三源一致→确定；两源→高；单源→中 |
| 社交互动 | 语音声纹(sim>0.5) + 对话内容 + 持续时间 | 用户发言+→高                      |
| 睡眠判断 | HR持续低位 + 长时间无交互 + 环境音      | 三源→高；两源→中                 |
| 用餐判断 | HR进食峰值 + 对话提及 + 环境音(Chewing) | 两源以上→高                       |

### 2. 上下文分析（核心改进）

**当前时段的每个理解段落，都必须结合前后文分析**：

- **前文**: "用户之前在做什么" → 活动是否自然延续
- **后文**: "用户之后做了什么" → 当前活动是否为过渡态
- **参考格式**: 在【原始数据摘要】中加"前后文"行，在【综合判断】中加"判断依据"行

> 示例：用户 10:00~10:30 在教室上课，10:30~10:35 HR突变 + GPS跳跃。只看10:30~10:35 会误判为"在移动"；结合前文可知是"课间换教室"。

### 3. 确信度标注规则

每个【综合判断】都**必须标注置信度**：

| 标注     | 适用场景                    | 示例                                           |
| -------- | --------------------------- | ---------------------------------------------- |
| 确定     | 多源证据一致指向唯一结论    | GPS在家+室内WiFi+环境音安静+屏幕锁屏→在家休息 |
| 高       | 2个以上数据源佐证，无矛盾   | GPS在校+PC有课堂笔记+环境音老师讲课→在上课    |
| 中       | 1个数据源佐证，其他数据缺失 | 仅有GPS坐标变化，其他数据空白→可能在移动      |
| 低       | 有部分数据但不足以确认      | 仅有一段环境语音→不确定用户参与程度           |
| 数据不足 | 该时段无数据或数据不完整    | 完全无感知数据→无法做出判断                   |

**置信度标注位置**：在【综合判断】末尾，格式为 `置信度: 确定/高/中/低/数据不足`

### 3b. 简洁性要求（理解文件精简规则）

**理解文件的职责是提炼，不是罗列。** 以下规则严格执行：

1. **HR分析禁逐分钟**：只写范围+趋势方向+异常点。禁止列出超过5个连续的HR值。
2. **综合判断禁止重复**：【综合判断】必须说原始数据摘要之外的新推断。如果只是对摘要的复述，说明你没有新的判断，应改为"结合前文后文判断活动自然延续"之类的一句话带过。
3. **关键观察可选**：没有新的交叉验证发现时，【关键观察】整节跳过，不留空。
4. **环境音标签不做故事**：只报告标签名+最高概率，不加"用户正在..."之类的演绎。
5. **App清单只概括**：不逐个列出App切换，而是概括模式（如"开发模式：Chrome↔OpenCode高频交替"、"娱乐模式：B站→网易云"）。
6. **避免健康建议**：不写"守护听力""建议休息"等系统无权给出的建议。只记录事实（如"耳机超时通知到达"）。
7. **通知联系人名不代表真实关系**：微信/QQ联系人备注名（如"姐姐""爸爸""老妈"等）不反映实际家庭关系。不要仅凭通知中的联系人名称推断人物关系。家庭关系的判断必须有语音内容或用户明确提及作为多源证据。
8. **身份不确定时用泛称**：无法通过声纹+内容+前后文确认说话人身份时，使用"有人""同行者""在场的人"等泛称。禁止给不确定的人安"舍友""同学""妈妈"等具体关系称呼。标注如"[身份不确定，泛称:有人]"。宁可模糊也不要错误。

### 4. 说话人归属规则

**说话人判断需结合声纹sim值 + ASR内容 + 对话前后文三者综合判断**，不能依赖单一信号。

以下是声纹置信度完整分级（来源：ai/AGENTS.md 原规范，已移入此处）：

| sim范围 | 含义 |
|:---:|------|
| > 0.5 | 用户本人清晰说话（测试短语/特意发音） |
| 0.3~0.5 | 用户日常对话范围（修复后系统） |
| < 0.3 | 环境/他人 |
| user_ratio | 有滑窗时优先于单条sim值 |

注意：voiceprint_speaker 字段始终显示"贺吸呼"（声纹库中唯一注册人），**不能**用于区分是否为用户说话。

优先级：

1. sim>0.5 → 用户本人（高置信）
2. 0.3~0.5 + 对话内容符合用户背景 → 用户日常对话
3. sim<0.3 + 对话内容与用户已知活动相关 → 可能为用户，标注"低置信"
4. sim<0.3 + 多人场景/嘈杂环境 → 结合前后文推断，标注置信度
5. 完全无语音数据 → 无法判断

当前格式（2026-06-24起）的事实文件中，语音行末尾附加 `窗值: 0.29→0.17→...`（最多8个滑窗值）。不再显示 `avg/min/max/user_ratio` 等统计字段。

- `窗值` 数组反映声纹在一段语音内的实时变化，值在0.3~0.5间波动为日常对话特征
- 滑窗数据在环境声/无语音段(sim<0.15)也生成，此时窗值全接近0
- 仍可参考旧rule：连续 0.3~0.5 波动→用户对话；持续 <0.15→环境音

> 注意：2026-06-24之前的数据文件可能仍包含旧格式 `[滑窗: avg=... min=... max=... user_ratio=0.x]`，按旧格式解析即可。

> **2026-06-18 声纹修复后新规律**: score_all()字典覆盖bug已修复、AAC压缩链路已对齐。修复后用户日常对话的sim值在 **0.3~0.5** 区间（而非>0.5）。测试/特意发音可达0.8~0.96。以下为更新规则：

| sim范围 | 归属标注               | 说明                                                                                   |
| :-----: | ---------------------- | -------------------------------------------------------------------------------------- |
|  > 0.5  | 用户本人（高置信）     | 测试短语/特意发音/特定发音场景                                                         |
| 0.3~0.5 | **用户日常对话** | 修复后系统下，这是用户自然对话的典型区间。除非多人场景有明确他人声音，否则优先归为用户 |
|  < 0.3  | 环境/他人              | 背景广播/他人对话/媒体内容。注：06-17及之前的旧数据因bug可能全局低配                   |

注意：宿舍/学校环境下 sim<0.5 的语音优先归为舍友。用户本人可能在场但基本不出声。

#### 4a. 声纹波动规律（2026-06-20 新增）

用户日常对话的声纹匹配值会在 0.3~0.5 区间内自然波动（标准差约 0.05），因为不同字词的发音特征不同——元音匹配高、辅音/爆破音可能偏低。**这种波动是"人在说话"的信号，不是噪声**。

判断方法：

- 连续语音段中 sim 在 0.3~0.5 间来回跳动 → **用户日常对话**
- 连续语音段中 sim 持续 < 0.15 且波动极小 → **环境音/他人**
- 测试短语 sim > 0.8 稳定 → **刻意发音/测试**

| 类型         | 典型范围 | 标准差 | 特征                 |
| ------------ | -------- | ------ | -------------------- |
| 用户日常对话 | 0.3~0.5  | ~0.05  | 有波动，字音不同导致 |
| 环境音/背景  | < 0.15   | ~0.02  | 几乎恒定             |
| 测试短语     | > 0.8    | <0.05  | 高且稳定             |

#### 4b. 旧数据全局低匹配（仅适用于06-17及之前的旧数据）

06-17及之前的语音因score_all()bug+AAC压缩链路未对齐，几乎所有语音sim<0.3。对于这些旧数据：

检查步骤：扫描该时段所有语音事件的 sim 值分布，如果全部 < 0.3，标记为"全局低匹配"
处理方式：添加 `> [声纹说明]` callout 说明，结合上下文判断说话人

### 4c. 用户HR参考范围

> 以下HR基线来源于06-07~06-18累计数据，分析HR时必须参考此基线，禁止无依据标注。

**数据质量须知（来源：ai/AGENTS.md 原规范，已移入此处）**：

- **HR数据交叉对比**：perception.jsonl 中的 HR 来自手机传感器快照，Gadgetbridge 被杀后台时读数会冻结。health.json 中的 HR 来自手环同步，是完整真实的记录。perception.jsonl 中 HR 长时间无波动（>30分钟）且 health.json 有不同数据时，以 health.json 为准。
- **GPS后台抑制--仅突变点可信**：Android 会对后台 App 的 GPS 定位进行抑制。GPS 突变点（坐标从A跳到B）= 成功定位一次，有参考价值。GPS 不变段（坐标长时间一致）= 数据未更新，不代表位置没变。分析位置时只看突变点。
- **ASR可能混入媒体内容**：用户观看电视剧/视频时，ASR 会将剧中对话识别为环境语音（sim<0.3），可能被误判为真实社交。sim<0.3 的环境语音若内容过于"剧情化"，标注"可能为媒体内容"。

数据来源于06-07~06-18累计数据。分析HR时必须参考此基线，**禁止无依据地标注"XX等于静坐"等错误判断**。

| 状态            | 典型HR范围      | 备注                        |
| --------------- | --------------- | --------------------------- |
| 深睡            | **45~48** | 稳定，health.json深睡期确认 |
| 浅睡/REM        | 49~65           | 正常睡眠周期波动            |
| 静坐(学习/工作) | **65~75** | 真正的静坐基线，不是57      |
| 闲聊/放松       | 68~78           | 日常交谈状态                |
| 室内走动        | 80~95           | 宿舍/家中走动               |
| 散步/慢走       | 90~100          | 户外步行                    |
| 进食            | 90~105          | 用餐中~餐后，峰值约105      |
| 公交/地铁站立   | 85~100          | 通勤状态                    |
| 情绪激动讨论    | 80~110          | 激烈对话/辩论/打电话        |
| 走路赶路        | 95~110          | 快步/负重                   |
| 起床过渡        | 49→80          | 从睡眠到日间水平需30~60分钟 |

> **health.json 延迟到达**: 手环断连后重连会补发整天数据。如果健康文件当时为空，标注"待补充"而非"缺失"，后续检查再补录。

> 57不是静坐的典型值，通常是从走动刚坐下尚未恢复、或刚睡醒半躺状态。

### 4d. ASR日语/韩语识别约束

事实文件中偶尔出现被识别为日语/韩语的语音文本。**绝大部分是ASR误识别**（环境噪声被误标为外语），不是用户在看日剧韩剧或讲外语。分析规则：

- 默认将日语/韩语文本标注为"环境声/ASR误识别"，置信度低
- 仅当存在多源证据时（PC窗口显示日剧、音乐播放日语歌、用户主动提及）才当作真实外语内容
- 不确定时标"可能为误识别"

### 6. 用户消息与任务触发解读

事实文件新增两个数据类型：

#### 用户消息（web_message）
```
- **用户消息**: HH:MM "消息内容"
```

- 来源：手机 app 内 WebView 聊天页面发送的消息（用户通过网页与 AI 对话时自动记录）
- 内容：用户发给 AI 的消息原文完整记录
- 置信度：**确定**（直接记录，无中间处理损失）
- 分析指引：
  - 直接作为用户明确意图和关注点的来源
  - 可与前后时段语音/AI回复对照，理解用户的工作或问题上下文
  - 如果用户在消息中提到了时间/地点/事件，可帮助确定其他数据的时间锚点
  - 消息内容可辅助判断用户情绪状态（语气、关键词）

#### 任务触发（task_trigger）
```
- **任务触发**: HH:MM [任务触发] 任务名
```

- 来源：AI 定时任务被系统触发时自动记录
- 格式：`[任务触发] 任务名`，任务名如"数据检查_14点"、"今日复盘"
- 置信度：**确定**
- 分析指引：
  - 标记系统自动执行任务的时刻，帮助区分用户行为 vs 系统行为
  - 任务触发前后的数据（如语音、屏幕状态）不应归因于用户主动行为
  - 可用于理解数据中断窗口（如任务跑脚本时可能无新感知数据）

> 两条规则：这两种数据**直接可信**（确定级置信度），无需交叉验证即可作为参考依据。但注意不要将系统任务触发的副作用（如脚本运行时的CPU占用）误判为用户活动。

### 5. 异常检测规则

以下情况需要标记异常，并**分析异常的可能原因**：

- 心率/体温等生理指标突变（跨时段相比），参考4c节HR范围判断是否异常
- 位置在短时间内大幅度跳跃（结合GPS前后文）
- 语音情绪明显异常（愤怒/哭泣/争吵等）
- 长时间无任何数据（超过2小时）
- 不符合该用户日常作息模式的行为
- 多源数据矛盾（如GPS显示在家但HR偏高、环境音嘈杂）

### 步骤 3: 交叉检查（验证理解是否符合已知用户信息）

写完所有段落后，做一次交叉检查，**防止写出与已知事实矛盾的判断**：

1. 回顾"步骤0"中读取的用户参考信息
2. 逐段检查写作的内容是否与已知信息矛盾：
   - 提到的人物是否用了正确的关系（家人/舍友/同学）？不知道的关系标"未知"
   - 提到的地点是否在已知范围内？用户没去过的地方要标注"疑似新地点"
   - 用户行为是否符合已知习惯？不符但数据充分就标"异常"
3. 发现矛盾立刻修正——**宁可加注"不确定"也不要写错误的确定性判断**

> 示例：用户参考.md 记录了"鹤城=用户本人"，如果理解文件里写"鹤城开车带用户去"，就是明显错误，应修正为"用户（鹤城）开车带某人去"。

### 步骤 4b: 写后验证（核心！防止edit未生效）

**写完所有时段后，必须验证edit是否真的写入了文件。**

操作方式：
1. 用 `read` 工具读取理解文件末尾（最后50行）
2. 确认新追加的时段（`## HH:MM~HH:MM`）确实出现在文件末尾
3. 如果发现追加内容不存在（常见失败原因：edit定位错误、文件路径不对、重复了已有内容等）：
   - 重新定位文件末尾，再次尝试 edit 追加
   - 如果仍失败，用 `read` 完整文件确认结构后再做第三次尝试
   - 三次失败后，在报告中说明"无法写入理解文件"
4. 确认新时段出现后再继续后续步骤

> 此步骤自2026-06-24 20:00故障后新增。那次子智能体声称完成但edit未生效，导致2小时数据丢失未被及时发现。写后验证可以杜绝此类问题。

## 禁止行为

- 不要修改事实文件（事实-全天.md）的任何内容
- 不要删除理解文件中已有的理解记录
- 不要调用Python脚本解析原始JSON数据（但允许用bash运行git命令提交）
- **不要写表面理解**——每个结论必须有数据支撑
- 数据不足时必须标注"数据不足"，**不准强行编造**
- 不要仅靠对话内容猜测说话人，但声纹sim值也不是绝对标准——**两者结合判断**
- 声纹清晰时(sim>0.5)：以声纹为主
- 声纹模糊时(sim<0.3且多人场景)：结合ASR内容和对话前后文推断谁在说话，标注置信度
- 宿舍/学校场景下，大量对话优先归为舍友，除非sim>0.5且内容与用户工作相关
- **不准只写结论不附证据**——综合判断必须有原始数据支撑
- **禁止发送任何通知**——子智能体不对外推送，主智能体负责通知控制
- **禁止给不确定身份的人安具体称呼**——身份不确定时用"有人""同行者"等泛称，标注不确定
