# 02 复盘子智能体

## 任务目标

基于理解记录做提炼总结，产出每日复盘报告。不接触原始感知数据，不做完整性验证。

## 核心原则

复盘的质量取决于理解记录的质量。如果理解记录有遗漏或错误，复盘也会有问题。复盘子智能体不做补漏工作 -- 那是数据检查子智能体（`01-data-check.md`）的职责。

## 触发方式

由主智能体通过 task() 传参触发：

- `date`: 日期文件夹名，如 2026-06-14
- `yesterday`: 昨日日期文件夹名，如 2026-06-13（用于跨日对比）

## 数据目录说明

| 路径 | 说明 |
|------|------|
| `ai/data/{date}/{date}-理解.md` | 当日理解记录（输入，带日期戳） |
| `ai/data/{yesterday}/{yesterday}-理解.md` | 昨日理解记录（跨日对比输入） |
| `ai/data/{date}/复盘-{date}.md` | 复盘报告（输出） |
| `ai/记忆/` | 记忆写入目标 |

## 工作流（8步）

### 步骤0: 加载用户上下文（建立认知基准）

**在读取理解文件之前，先了解用户的背景信息。**

1. 用 `read` 工具读取 `ai/记忆/用户参考.md` —— 了解用户的人际关系（家人/同学/舍友/朋友/恋人）、常去地点、偏好习惯
2. 用 `read` 工具读取最近2~3天的理解文件（如 `{date-1}-理解.md`、`{date-2}-理解.md`），刷新近期事件脉络
3. 建立"用户画像"：用户的家人有谁？好友有谁？常去哪些地方？有什么习惯？

> 有了这个基准，才能发现理解文件中的事实错误。

### 步骤1: 读{date}-理解.md全文

用 `read` 工具读取 `ai/data/{date}/{date}-理解.md` 文件的全部内容。通读所有时段的理解记录，掌握全天概貌。

**读的同时对照步骤0的用户画像**，标注可能的错误：
- 人物关系描述是否和已知一致？比如"(占位符)"不是"用户的朋友"而是"用户本人"
- 地点描述是否合理？用户没去过的地需标"疑似新地点"
- 行为习惯是否和用户一致？

不要读原始 `perception.jsonl` 或 `health.json` -- 那是数据检查子智能体的工作。

### 步骤1b: 学习进度回顾（新增）

读取 `ai/data/学习进度/{date}.md`（如有），将学习进度纳入复盘：

1. 今天的学习任务完成情况（词组/英语/专业课等）
2. 与昨天的学习对比（进步/退步）
3. 距离考试的时间提醒（如果进度落后，在报告中温和标注）
4. 如有两周计划，检查是否按计划推进

将学习进度写入复盘报告中的"学习进度"章节。

### 步骤1c: 合理性检查（检查推断错误和前后矛盾）

读完全文后，做一次系统性检查，**标注理解文件中的可疑推断和矛盾**：

1. **推断过度检查**：逐段看综合判断，是否有证据不足的高置信结论
   - 标注示例：`> [检查] 13:00段"用户在学习"标注为"确定"但仅有PC窗口证据，缺乏语音/HR佐证，应为"中"`
2. **身份猜测检查**：是否有给不确定的人安了具体称呼（姐姐/舍友/同学等）
   - 改为泛称，在事实修正章节记录
3. **时间线矛盾检查**：同一人物/活动在不同时段描述是否一致
   - 如"10:00在教室"和"10:05在校外"需核实是否有GPS突变
4. **主角归属检查**：生日/庆祝/聚会等活动是否默认主角是用户
   - 确认语音内容中是否有"你生日""他生日"等第三人称指代

发现问题后用 `edit` 修改理解文件，在修正处标注 `> [修正]`。在复盘报告的事实修正章节记录。

### 步骤2: 提取关键事件

从理解记录中提取 3~5 个关键事件。每个事件包含：

- 时间: 事件发生的时段
- 位置: 事件发生的地点
- 一句话描述: 发生了什么
- 置信度: 确定/高/中/低

区分"用户主动行为"和"被动事件"：

| 类别 | 定义 | 示例 |
|------|------|------|
| 用户主动行为 | 用户发起的动作 | 上课、出门、做饭、学习 |
| 被动事件 | 外部触发的事件 | 闹钟响、收到通知、室友回来 |

### 步骤3: 对比昨日找变化

用 `read` 工具读取 `ai/data/{yesterday}/{yesterday}-理解.md` 的全文。从以下维度对比：

- 作息变化: 起床时间、睡觉时间、午休时间的变化
- 活动模式变化: 主要活动类型的变化（上课/自习/外出等）
- 位置变化: 主要停留地点的变化

每项对比标注置信度。如果理解记录中数据不足以判断，写"数据不足以对比"。

### 步骤4: 异常检测

从理解记录中检查以下异常情况。注意：异常检测依赖理解记录中标注的异常标记，不做独立数据分析。

- 心率突变: 理解记录中提及的心率比前一时段变化 >15bpm
- 位置异常: 凌晨时段出现在非睡眠地点
- 情绪异常: 理解记录中标注了愤怒/哭泣等异常情绪
- 长时间数据空缺: 理解记录中标注了超过 3 小时无数据
- 其他异常: 理解记录中标注的其他异常

每个检测到的异常写一条记录：时间、异常类型、描述、置信度。

### 步骤4b: 事实修正（修复理解文件中的明显错误）

如果在步骤1中发现理解文件与已知的用户信息矛盾，**直接修正理解文件**：

1. 用 `edit` 工具修改 `{date}-理解.md` 中的错误内容
2. 在修正处标注 `> [修正]` 注明修正原因
3. 在复盘报告的"事实修正"章节记录修正项

> 修正示例：
> ```
> 【源数据】GPS:(占位符),(占位符) | voice: "(占位符)开车带我去"
> 【分析】18:30 和舍友(占位符)一起开车去餐厅 ← 错误
> 【修正】18:30 用户本人(占位符)开车带人去餐厅
> ```

不修改原始数据文件（perception.jsonl / health.json）。

### 步骤5: 记忆收割

从理解记录中发现值得写入长期记忆的信息。记忆收割的目标是捕捉那些"以后可能有用"的信息。

| 类型 | 收割条件 | 写入位置 |
|------|----------|----------|
| 新地点 | 出现未在已有的记忆文件中记录过的 GPS 坐标或地点名 | 更新 `ai/记忆/用户参考.md` 中的地点记录 |
| 新偏好 | 饮食习惯、娱乐方式、作息习惯等初次或持续出现 | 更新 `ai/记忆/用户参考.md` 中的偏好记录 |
| 新项目 | 用户提到的学习项目、工作项目、兴趣方向 | 更新 `ai/记忆/用户参考.md` 中的项目记录，或创建新文件 |
| 新社交关系 | 初次或反复出现的人物名称 | 更新 `ai/记忆/用户参考.md` 中的社交关系记录 |
| 健康变化 | 持续的生理指标变化（非临时波动） | 更新 `ai/记忆/用户参考.md` 中的健康基线 |
| 新经验教训 | 用户踩坑或成功经验（可复用的） | 写入 `ai/记忆/经验/` 下新文件，并更新 `ai/记忆/经验记录.md` 索引 |

### 步骤6: 写复盘报告

将复盘结果写入 `ai/data/{date}/复盘-{date}.md`。如文件已存在则覆盖。

报告格式：

```markdown
---
date: {date}
tags: [复盘报告, 感知分析, 跨日对比]
aliases:
  - {date} 复盘报告
links:
  - [[{date}-理解|当日理解]]
  - [[日志-{date}|对话日志]]
  - [[复盘-{yesterday}|昨日复盘]]
---

# {date} 复盘报告

## 关键事件

| 时间 | 类别 | 事件 | 置信度 |
|------|------|------|--------|
| HH:MM | 主动/被动 | 事件描述 | 确定/高/中/低 |
| HH:MM | 主动/被动 | 事件描述 | 确定/高/中/低 |
| HH:MM | 主动/被动 | 事件描述 | 确定/高/中/低 |

## 对比昨日

- 作息: 变化/一致/异常（简要说明 + 置信度）
- 活动: 变化/一致/无对比数据（简要说明 + 置信度）
- 位置: 变化/一致/无对比数据（简要说明 + 置信度）

## 异常检测

- 有/无异常

如有异常则逐条列出：
1. 时间: 异常类型 — 描述 [置信度]

## 学习进度

- 今日完成：词组X个 / 英语X讲 / 专业课X讲 / 阅读X篇
- 计划对比：按计划推进中 / 略微落后 / 大幅偏离
- 与昨日对比：进步 / 持平 / 退步
- 考试倒计时：距离2026年3月22日还有X天

## 记忆收割

列出本次收割/更新的记忆条目：

- 名称: xxx（类型: 新地点）
- 名称: xxx（类型: 新偏好）

## 事实修正

列出本次修正的理解文件错误（如有，无则写"无修正"）：

| 时间 | 原内容 | 修正后 | 修正原因 |
|------|--------|--------|----------|
| HH:MM | 错误描述 | 正确描述 | 依据用户参考.md |

## 关联文件

- 当日理解: [[{date}-理解|当日理解记录]]
- 昨日复盘: [[复盘-{yesterday}|昨日复盘]]
- 当日日志: [[日志-{date}|当日日志]]
- 相关经验: 如有关联经验文件，列出 [[../../记忆/经验/xxx]]

## 自检结果

- 通过/未通过
```

### 步骤7: 自检

完成所有步骤后，逐项核对自检清单：

- [ ] 步骤0：已读取用户参考.md，建立用户认知基准
- [ ] {date}-理解.md全文已读，并与用户画像对照检查
- [ ] **合理性检查已完成**——推断过度/身份猜测/时间线矛盾/主角归属四项全部检查
- [ ] 发现的问题已修正并记录在事实修正章节
- [ ] 关键事件已提取（3~5个）
- [ ] 跨日对比已完成
- [ ] 异常检测已完成
- [ ] 学习进度已回顾（步骤1b），报告中有对应的进度章节
- [ ] 发现的理解文件错误已修正（步骤4b），报告中有对应记录；如无修正则标注"无修正"
- [ ] 记忆已收割
- [ ] 报告已写入 `ai/data/{date}/复盘-{date}.md`
- [ ] 报告 frontmatter 包含 `links:` 字段，引用了当日理解、日志、昨日复盘
- [ ] 报告末尾包含 `## 关联文件` 章节，列出了相关文件的 wiki 链接
- [ ] 报告包含 `## 事实修正` 章节（无修正也保留）
- [ ] 记忆收割的文件也添加了 wiki 回链到复盘报告
- [ ] 没有修改原始数据（perception.jsonl / health.json）
- [ ] 理解文件的修改仅限于事实修正（步骤4b），没有做其他修改
- [ ] 报告格式符合规范

## Obsidian 关联链接操作指南

在复盘报告中添加 wiki 链接时，注意路径规则：

| 目标文件 | 从 `data/{date}/` 出发的相对路径 | 示例 |
|---------|--------------------------------|------|
| 当日理解 | `{date}-理解`（同一目录） | `[[2026-06-17-理解]]` |
| 昨日复盘 | `复盘-{yesterday}.md`（需跳出目录） | `../{yesterday}/复盘-{yesterday}.md` 或 `复盘-{yesterday}` |
| 当日日志 | `日志-{date}`（同一目录） | `[[日志-2026-06-17|当日日志]]` |
| 经验文件 | `../../记忆/经验/xxx.md` | `[[../../记忆/经验/HR数据交叉对比.md]]` |

> 使用 Obsidian 的 `[[链接|显示名]]` 语法，让链接可读性好。

## 明确不做

- **不做完整性验证**: 数据是否完整由数据检查子智能体（`01-data-check.md`）和 `check_coverage.py` 脚本负责
- **不逐段回查原始数据**: 复盘完全基于理解记录，不读 `perception.jsonl` 或 `health.json`
- **不修改工具脚本**: 不改 `ai/analysis/` 下的脚本
- **不主动修改理解文件**: 唯一例外——步骤4b中发现的明显事实错误可以修正。修正时必须标注 `[修正]` 说明原因

## 常见陷阱

| 陷阱 | 后果 | 预防 |
|------|------|------|
| 理解记录有遗漏 | 复盘缺失关键事件 | 依赖数据检查子智能体补漏，不复核原始数据 |
| 理解记录置信度低 | 复盘结论不可靠 | 对低置信度的理解在复盘中标注"待确认" |
| 跨日对比时数据不足 | 对比结论不准确 | 无数据时标注"数据不足以对比" |
| 记忆收割过泛 | 记忆文件膨胀 | 只收割可复用的信息，不记临时事件 |

## 提交 git

复盘报告写完后运行：

```bash
git add ai/ && git commit -m "今日复盘完成"
```

## 发送通知

复盘完成后用 `screen-mcp_send_notification` 发送通知，格式：

- 标题: "复盘完成"
- 内容: 一句话总结（如"昨天你睡了X小时、做了X件主要的事、发现了X个值得注意的点"），不超过80字
